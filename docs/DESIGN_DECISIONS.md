# Design Decisions & Known Limitations

This document records key architectural choices and known limitations of the `tiled-ip` framework.

---

## 1. Block Isolation vs. Stitching

**Decision:** `AXI4StreamLiteBase` creates a fresh `pyrtl.Block()` per instance to avoid wire-name collisions.

**Rationale:** Each IP instance needs its own isolated PyRTL block so that wire names don't clash when multiple instances are created. This is essential for unit testing each IP in isolation.

**Stitching requirement:** To stitch IPs together, all IPs must be instantiated in the **same** block. IPs that expose a `block=` parameter (GEMM, Softmax, MemRouter, TemporalGEMM, StatefulSoftmax, MultiBankBRAM) pass it through directly. IPs that do not (Norm, Activation, ALU, StatefulNorm) require a `pyrtl.Block` monkey-patch during instantiation.

**Automatic handling:** The code generated by `generate_subgraph.py` handles this automatically by monkey-patching `pyrtl.Block` before instantiating IPs that don't accept a block parameter.

---

## 2. Simulation Wrappers

**Decision:** Many cores keep their AXI4-Stream-Lite ports as plain `WireVector` so they can be driven by upstream IPs.

**Rationale:** This allows direct stitching via `<<=` without going through Input/Output wrappers.

**Impact:** Standalone and compound testbenches must create wrapper `Input`/`Output` wires and connect them with `<<=` to drive/receive data from the simulation.

---

## 3. Memory Sync Checks

**Decision:** Some LUT ROMs are accessed combinationally (address depends on upstream logic).

**Rationale:** The LUT-based implementations (Softmax, Norm, Activation) use ROMs for fixed-point approximations. These are read combinationally because the address is derived from data values.

**Problem:** PyRTL's `sanity_check_memory_sync` can flag this as an error in shared blocks because the address logic and memory access form a combinational path.

**Workaround:** Testbenches disable the check on the shared block instance:
```python
block.sanity_check_memory_sync = lambda wire_src_dict=None: None
```

---

## 4. Gated-GELU Out of Scope

**Decision:** The `ActivationCore` only has one input stream.

**Rationale:** A true gated-GELU (SiLU-style) requires a second stream: `gate = sigmoid(x)` and `value = x`. Implementing this would require a second input port and multiplexer logic.

**Current support:** Only standard GELU (tanh approximation) and ReLU are supported.

---

## 5. Fan-In Support

**Decision:** Fan-in (N→1) is supported only for IPs with a `data_in_b` port (e.g., `ALUCore`).

**Rationale:** Only the ALU has a second input port for element-wise operations. Implementing fan-in for all IPs would require generic multiplexing logic that complicates the design.

**Implementation:** The Stitcher wires the first upstream to `data_in`, the second to `data_in_b`, and `valid_in` = AND of all upstream `valid_out` signals. Fan-in on IPs without `data_in_b` raises `ValueError`.

---

## 6. `last_in` / `last_out` Protocol

**Decision:** Added to `AXI4StreamLiteBase` as lazy properties (created on first access).

**Rationale:** This preserves backward compatibility with existing IPs that do not use burst markers.

**Behavior:**
- `last_in` / `last_out` are created on first access, not during `__init__`
- Temporal cores deassert `ready_out` during `COMPUTE` / `EMIT` phases to prevent upstream data from overwriting internal state
- Block isolation: `last_in` and `last_out` are created inside the IP's own block, so stitching does not leak wires across blocks

---

## 7. StreamShape System

**Decision:** Use `StreamShape(N, T)` to track tensor dimensions and auto-insert `StreamShapeAdapter` for T mismatches.

**Rationale:**
- `N` (total elements) is preserved across transformations
- `T` (elements per beat) can vary per IP based on spatial tiling
- The Stitcher can automatically resolve T mismatches by inserting adapter IPs
- Shape propagation is validated topologically to catch errors early

**Validation:**
- `N % T == 0` must hold for all shapes
- Source `N` must be divisible by destination tiling param

---

## 8. Temporal GEMM Uses `last_in` for Accumulation Control

**Decision:** All temporal GEMMs have `accum_in` tied high and `emit_in` tied low; accumulation is controlled entirely by `last_in`.

**Rationale:** This simplifies the external control interface. When `last_in` is asserted on the final beat of a multi-beat packet, the core transitions from accumulation to emission.

---

## 9. Fixed-Point LUT Approximations

**Decision:** Use pre-computed ROM LUTs for expensive operations (exp, 1/√x, tanh, gelu).

**Rationale:** True exponentiation and division are hostile to FPGAs. LUTs provide deterministic, fast approximations that fit well in BRAM.

**Tolerance:** The fixed-point approximations introduce small errors (documented in [TESTING.md](TESTING.md#55-quantization-tolerance)).

---

## Known Limitations

1. **Gated-GELU not supported** — requires second input stream
2. **Single-block stitching only** — all IPs must share the same PyRTL block
3. **Manual `last_in` wiring** — some temporal core `last_in` signals require manual connection after `stitcher.build()`
4. **Side ports not wrapped** — `ALU.op_code`, `GEMM.weight_in`, `TemporalGEMM.accum_in` etc. must be driven externally
5. **Fixed-point only** — no floating-point support in hardware
